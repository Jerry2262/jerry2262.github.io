# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A static, no-build personal site hosted on GitHub Pages at `jerry2262.github.io`. The repo doubles as an app portal: the homepage links to independent browser apps, each living in its own directory under `apps/`. There is no package manager, bundler, framework, or test runner — every app is plain HTML/CSS/JS served as static files.

## Commands

There is no build or test step. To develop, open files directly in a browser, or serve the repo root:

```bash
python3 -m http.server 8000   # then visit http://localhost:8000/
```

The one tooling script is the secret-page encryptor (requires Node):

```bash
node scripts/encrypt-secret.js --init        # create local private/page.html + password.txt from the example
node scripts/encrypt-secret.js               # encrypt private/page.html into apps/secret/payload.js
node scripts/encrypt-secret.js --force       # --init that overwrites existing private files
```

The encryptor refuses to write outside the repo and enforces PBKDF2 iterations ≥ 100000. The password is read from `apps/secret/private/password.txt` or the `SECRET_PAGE_PASSWORD` env var.

## Architecture

### Homepage as entry, apps as the body

`index.html` + `styles.css` form the landing page. Apps are linked as cards in `.feature-grid`. When adding an app: create `apps/<name>/` with its own `index.html` (+ `style.css`/`game.js` as needed), then add a card in `index.html`. The grid column count in `styles.css` (`.feature-grid` `grid-template-columns`) must match the number of cards, and there is a mid-width breakpoint that drops it to 2 columns — adjust both when adding/removing apps.

Each app is fully self-contained (own HTML, CSS, JS) with no shared runtime; they link back to the homepage via `../../index.html`.

### Secret app (encrypted page pattern)

This app is unlike the others — it ships only ciphertext and decrypts at runtime. Understanding it requires reading several files together:

- `apps/secret/private/page.html` + `private/password.txt` — **local only, git-ignored** (see `.gitignore`). The plaintext page and its password never enter version control.
- `scripts/encrypt-secret.js` — Node-side: derives an AES-256-GCM key from the password via PBKDF2(SHA-256, 210000 iterations) and writes the ciphertext + KDF params to `apps/secret/payload.js` as a `SECRET_PAGE_PAYLOAD` global.
- `apps/secret/payload.js` — the committed ciphertext (regenerated by the script, not hand-edited).
- `apps/secret/secret-loader.js` — browser-side: uses the Web Crypto API to re-derive the key from the user's password, attempts AES-GCM decrypt, and on success replaces the document via `document.open()/write()/close()`. A wrong password fails the GCM auth tag and the page stays on the unlock panel.
- `apps/secret/secret-source.example.html` — the template copied by `--init` to seed `private/page.html`.

So the flow is: edit `private/page.html` locally → run the encryptor → commit the regenerated `payload.js`. The KDF params in `payload.js` (salt, iv, iterations) must be consumed verbatim by the loader; the two files are a matched pair.

### Texas Hold'em AI provider seam

`apps/texas-holdem/` separates the game engine (`engine.js`), UI (`app.js`), and AI (`ai-provider.js`). Per `INTERFACE.md`, the AI provider exposes `chooseAction(context)` returning `{type, raiseTo?}` with types `fold|check|call|raise`. This seam exists so a future WebAssembly module can replace `ai-provider.js` alone without touching the UI. When changing AI behavior, implement against this contract rather than coupling into `app.js`.
